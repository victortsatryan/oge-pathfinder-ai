# Pathfinder — Архитектура продукта

> Шаг 0. Архитектурная рамка. Этот документ описывает целевую модель продукта
> и принципы, по которым будут развиваться следующие шаги. Текущий код
> (ОГЭ-MVP) **сохраняется** и постепенно переезжает в новую структуру —
> ломающих изменений на этом шаге нет.

---

## 1. Что такое продукт

**Pathfinder** — платформа для построения **индивидуального учебного
маршрута** ученика по любым предметам, программам и целям.

Подготовка к ОГЭ — это **один из сценариев** (`exam_type = "OGE"`), а не
основа архитектуры. В дальнейшем поддерживаются: ЕГЭ, школьная программа,
олимпиады, внешкольные предметы, профессиональные навыки, языки и т.д.

Метафора продукта — **карта знаний + мастерская + исследовательский архив**.
Пользователь не «проходит курс», а исследует территорию предмета,
отмечает препятствия и строит маршрут.

---

## 2. Главная сущность: Student Profile

Главная сущность продукта — **не календарь, не урок, не диагностика и не
ассистент**. Главная сущность — **профиль ученика** (`Student Profile`).

Профиль ученика — центральный источник правды. В будущем он хранит:

- учебные цели (`goals`: экзамен, балл, дедлайн, школьная программа, …);
- выбранные предметы и уровни;
- прогресс по темам и подтемам;
- слабые темы и типичные ошибки;
- результаты диагностик;
- историю занятий и выполненных заданий;
- AI-рекомендации;
- связь с одним или несколькими учителями.

Все остальные модули (календарь, диагностика, занятия, аналитика,
ассистент) **читают и обновляют** профиль ученика, а не живут отдельно.

---

## 3. Главная цепочка данных

```
Account
  └── User Profile
        └── Student Profile
              ├── Goals / Exam type
              ├── Subjects
              │     └── Topics / Subtopics
              │           ├── Diagnostics results
              │           ├── Progress
              │           └── Weak spots
              ├── Materials (theory, video, tasks)
              ├── Lessons
              ├── Learning Path  ──►  Calendar
              ├── Analytics
              └── AI Assistant (читает контекст профиля)
```

Любая фича привязывается к этой цепочке. Если экран не вписывается в
неё — это сигнал пересмотреть его место в продукте.

## 4. Модули

Целевая модульная структура. На шаге 0 заводятся **папки-заготовки** под
будущие модули; код переезжает постепенно.

| Модуль | Назначение | Сейчас |
|---|---|---|
| **auth** | Авторизация, сессия, восстановление | `src/hooks/use-auth.tsx`, `src/routes/login.tsx` (скрыт) |
| **account** | Профиль аккаунта, роль | `profiles` таблица, `role.functions.ts` |
| **student-profile** | Учебный профиль ученика — ядро продукта | будет на шаге 1 |
| **subjects** | Справочник предметов | `subjects` таблица |
| **topics** | Темы и подтемы внутри предметов | `topics` таблица |
| **diagnostics** | Входная, недельная и кастомная диагностика | `oge-diagnostic.functions.ts`, `diagnostic_sessions` |
| **materials** | Теория, видео, задания, тесты | `content_resources`, `learning_sources`, `tasks` |
| **lessons** | Учебные занятия как сборка темы + материалов + заданий | `lessons`, `oge-lesson*.functions.ts` |
| **learning-path** | Индивидуальный маршрут | `study_plans` (заготовка) |
| **calendar** | Календарь, построенный из learning-path | `oge-mvp-app.tsx` (текущий MVP) |
| **analytics** | Прогресс, слабые темы, ошибки, динамика | `task_attempts`, `profile.functions.ts → getMyProgress` |
| **assistant** | AI поверх данных ученика | `oge-assistant.functions.ts`, `assistant_*` таблицы |
| **teacher-relations** | Связь учитель ↔ ученик | `students` таблица (архитектурная заготовка) |

### Целевая структура папок (заводится сейчас, наполняется постепенно)

```
src/
  modules/
    student-profile/   # ядро (шаг 1)
    subjects/
    topics/
    diagnostics/
    materials/
    lessons/
    learning-path/
    analytics/
    assistant/
    teacher-relations/
  routes/              # роуты остаются здесь
  components/          # UI компоненты
  lib/                 # текущий код (oge-*), переезжает постепенно
  integrations/        # supabase, lovable — не трогаем
```

Каждая папка модуля имеет `README.md` с назначением модуля и ссылкой
на этот документ. Реальный код переезжает по мере выполнения следующих
шагов — старые `src/lib/oge-*.ts` пока остаются на месте.

## 5. Роли пользователей

- **student** — приоритетная роль, фокус шагов 1–N.
- **teacher** — учитывается архитектурно (таблица `students`,
  кабинет `/teacher/*`), не развивается активно сейчас.
- **admin** — зарезервировано (через `user_roles` + `has_role`).

Бизнес-роль `student/teacher` хранится в `profiles.role` (выбор сценария).
Привилегии `admin` — только через таблицу `user_roles` и функцию
`has_role()` (по правилам безопасности проекта).

Связи (будущее):
- один ученик ↔ один или несколько учителей;
- один учитель ↔ много учеников.

---

## 6. Связи между модулями

**Профиль ученика — центр.** Из него читают контекст:

- **Диагностика** → пишет результаты в профиль → обновляет
  прогресс/слабые темы → триггерит рекомендации → перестраивает маршрут.
- **Календарь** строится **не вручную**, а из `learning-path`:
  `learning path → planned items → lessons → calendar`.
- **Занятие** связано с: учеником, предметом, темой, материалами,
  заданиями, результатами выполнения.
- **Материалы** связаны с: предметами, темами, уровнем сложности,
  источниками, типом материала.
- **Ассистент** получает контекст из: профиля ученика, выбранного
  предмета/темы, диагностик, ошибок, материалов, текущего занятия,
  маршрута. **Ассистент — не отдельный чат-бот.**

---

## 7. Принцип работы AI

AI **не является источником правды**. Источник правды — структура
данных продукта (профиль, темы, диагностика, прогресс, материалы,
задания, занятия).

AI:

- объясняет;
- рекомендует следующий шаг;
- подбирает материалы;
- помогает анализировать ошибки;
- помогает собирать занятия;
- помогает преподавателю и ученику.

AI **не** заменяет педагогическую логику и не хранит состояние
обучения вне таблиц продукта.

## 8. Терминология и нейминг

Постепенно отвязываем нейминг от ОГЭ. Новые сущности **не должны**
называться через `oge-*`. Используем универсальные:

| Старое (остаётся) | Новое (для нового кода) |
|---|---|
| `oge-ai` | `assistant` |
| `oge-diagnostic` | `diagnostics` |
| `oge-mvp` / `oge-calendar` | `learning-path` / `calendar` |
| `oge-lesson` | `lessons` |

ОГЭ становится **параметром** (`exam_type`, `program`), а не корнем
системы. Существующие файлы `src/lib/oge-*.ts` **не переименовываем
прямо сейчас** — они продолжают работать, миграция произойдёт по мере
переезда логики в `src/modules/*`.

## 10. Следующие шаги

1. **Шаг 1 — Ядро Student Profile.** Таблица `student_profiles`
   (или расширение `profiles`), учебные цели, выбранные предметы,
   карта прогресса по предметам/темам. Серверные функции
   `student-profile/*.functions.ts`.
2. **Шаг 2 — Темы и подтемы.** Наполнение `topics`, привязка
   прогресса и слабых мест к темам.
3. **Шаг 3 — Диагностика, пишущая в профиль.** Переезд
   `oge-diagnostic` в `modules/diagnostics`, запись результатов
   в профиль ученика.
4. **Шаг 4 — Learning Path.** Построение маршрута из профиля,
   календарь читает маршрут.
5. **Шаг 5 — Материалы и занятия как сборка.** Связь
   материал ↔ тема ↔ занятие.
6. **Шаг 6 — Аналитика и AI поверх профиля.**
7. **Шаг 7 — Teacher relations.** Полноценный кабинет учителя
   с доступом к профилям учеников.

## 11. Ограничения шага 0

На этом шаге **намеренно не делается**:

- переписывание базы данных;
- удаление текущих экранов и ОГЭ-разделов;
- переделка дизайна;
- загрузка материалов;
- полноценный кабинет учителя;
- сложный AI;
- изменение routing.

Цель шага 0 — **рамка**, а не реализация.
